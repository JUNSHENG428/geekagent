推荐第一阶段插件

先不要一次性接太多。MVP 阶段的目标是确认谁在用、哪里卡住、谁愿意付费。

  • Supabase：Postgres 数据库、认证、RLS、Edge Functions，承接订阅、评论、配置事件和诊断工单。
  • Resend：发送模板更新、诊断预约确认、购买交付邮件。
  • Stripe / Lemon Squeezy：出售模板包和诊断服务，先用 Payment Link 降低开发量。
  • PostHog：分析生成、下载、诊断、订阅、购买点击等转化事件。
  • Giscus：如果暂时不做自建评论，可用 GitHub Discussions 承接教程页评论。
Decision

我建议先接 Supabase + Resend

原因很简单：这两个能立刻把邮箱、配置反馈、诊断线索和后续模板交付串起来。支付可以等模板包内容稳定后再接。

插件优先级

优先级插件/服务接它解决什么暂时不接的风险
P0Supabase数据库、RLS、评论、订阅、诊断工单用户反馈散落在邮箱和 localStorage
P0Resend邮件订阅、购买交付、诊断确认无法自动触达高意图用户
P1PostHog 或 Vercel Analytics看清生成、诊断、购买点击漏斗不知道哪些页面真正转化
P1Stripe / Lemon Squeezy模板包和诊断付费商业化只能靠手动收款
P2Giscus无需自建审核后台的教程评论社区讨论沉淀慢
P2Cloudflare Turnstile保护评论、订阅、诊断表单表单开放后容易被垃圾提交
P3Algolia / Pagefind教程和报错库搜索内容多了以后查找效率下降

接入顺序

  1. 建 Supabase 项目，执行 docs/database-schema.sql。
  2. 把 newsletter、comments、diagnose、wizard 事件接到 Edge Function。
  3. 接 Resend，把订阅和诊断预约自动回信。
  4. 模板包稳定后再接 Stripe / Lemon Squeezy。
  5. 最后补 PostHog 漏斗和 Turnstile 防滥用。

前端改造点

  • 保留当前纯前端 fallback，数据库失败时不阻塞核心工具。
  • 所有用户可识别数据必须有同意文案和隐私说明。
  • 评论默认进入 pending 状态，审核后展示。
  • 配置生成记录默认匿名化，不保存原始敏感路径。
